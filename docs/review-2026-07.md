# HEALRISE — Gesamtreview (03.07.2026)

Konsolidiertes Ergebnis aus drei Review-Strängen (Frontend F1–F36, Backend B1–B11, Build/PWA/Infra I1–I15)
plus Abgleich untereinander. Referenz für den Umsetzungsplan in `docs/plan/`.

## Wichtige Korrekturen aus dem Abgleich der Reviews

Das Frontend-Review nahm an, das Gating sei rein clientseitig (ursprüngliches F2/F3). Das Backend-Review zeigt:
**Es existiert ein Custom-Controller mit serverseitigem Gating** (`strapi/src/api/program/controllers/program.ts`) — aber:

1. `find` filtert gesperrte Programme **komplett weg** → die Lock-Overlay/Upgrade-UX des Frontends kann nie
   erscheinen; der Goldstandard (U-Ableitung: „Upgrade-Pfad direkt aus gesperrten Inhalten heraus") ist damit
   unerfüllbar. Richtig wäre: Metadaten aller Programme ausliefern, nur `body`/`video_url` bei fehlender
   Berechtigung strippen.
2. `findOne` prüft `result.data.attributes?.plan_required` — **Strapi 5 liefert flache Objekte** (kein
   `attributes`) → der Check läuft immer gegen `undefined` → default `freebie` → **Gating in findOne wirkungslos**
   (Premium-Body per documentId abrufbar).
3. `updateMe` wird als Permission aktiviert, **existiert aber in Strapi 5 users-permissions nicht**; das Frontend
   ruft `PUT /api/users/:id` (= Action `user.update`, NICHT aktiviert) → Profil-Edit & Passwortänderung liefern
   vermutlich 403 (Laufzeit-Verifikation ausstehend). Positiv: der befürchtete Selbst-Upgrade über `plan`-Feld
   (F3) ist dadurch aktuell NICHT möglich — würde aber sofort aufgehen, wenn jemand `user.update` freischaltet.

## Architektur-Ist-Zustand (Kurzfassung)

- **Frontend:** React 19 + Vite 8 + vite-plugin-pwa, axios, react-router 7; kein TS, keine Tests; Base `/healrise/app/`.
  Seiten: Login, Dashboard, Pläne (Wochen/Tage + Checkboxen), Programm-Detail (CMS-HTML), Konto, Upgrade (mailto).
  Auth: Strapi-JWT in localStorage, axios-Interceptors, Route-Guards. Fortschritt: nur localStorage.
- **Backend:** Strapi 5.46, Postgres, ein Content-Type `program` (+ User-Erweiterung `plan`, `plan_purchased_at`,
  `plan_expires_at`), Custom-Controller mit (fehlerhaftem) Gating, Bootstrap setzt Permissions + seedet
  Testuser/Demo-Content. Kein Payments-, kein Progress-Modell.
- **Deployment:** services.frigew.ski/healrise/ (Landing = dist/, App = app/dist, CMS = Strapi auf 127.0.0.1:9130
  hinter Reverse Proxy). Kein Git, keine CI, keine Tests.
